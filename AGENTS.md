# AGENTS.md

## Огляд проєкту

`javahandbook` - це Spring Boot вебзастосунок для навчального довідника з Java.

Основна функціональність:

- Публічні сторінки зі статтями про Java.
- Ліве бокове меню, згруповане за розділами.
- Сторінки статей з HTML-контентом.
- Публічна система коментарів із відповідями.
- Модерація коментарів в адмін-панелі.
- Базовий підрахунок відвідувань.
- Email-сповіщення адміну про нові коментарі.

## Технологічний стек

- Java - цільову версію завжди перевіряти в `pom.xml`.
- Maven.
- Spring Boot.
- Spring MVC.
- Spring Data JPA.
- Spring Security.
- Spring Mail.
- Thymeleaf.
- MySQL.
- Звичайні CSS і JavaScript без окремого frontend-фреймворку.

## Основна структура

Важливі директорії:

- `src/main/java/com/example/javahandbook/controller` - MVC-контролери.
- `src/main/java/com/example/javahandbook/service` - бізнес-логіка.
- `src/main/java/com/example/javahandbook/entity` - JPA entity-класи.
- `src/main/java/com/example/javahandbook/repository` - Spring Data репозиторії.
- `src/main/java/com/example/javahandbook/dto` - DTO для форм і відображення.
- `src/main/resources/templates` - Thymeleaf-сторінки.
- `src/main/resources/templates/fragments` - спільні Thymeleaf-фрагменти.
- `src/main/resources/static/css` - стилі.
- `src/main/resources/static/js` - frontend-скрипти.
- `src/main/resources/application.properties` - локальна конфігурація.

## Основні маршрути

Публічні маршрути:

- `GET /` - головна сторінка.
- `GET /articles/{slug}` - сторінка статті.
- `POST /articles/{slug}/comments` - надсилання коментаря або відповіді.
- `GET /login` - сторінка входу в адмін-панель.

Адмін-маршрути:

- `GET /admin` - головна сторінка адмін-панелі.
- `GET /admin/articles` - керування статтями.
- `GET /admin/articles/edit/{slug}` - редагування статті.
- `POST /admin/articles` - створення статті.
- `POST /admin/articles/update/{slug}` - оновлення статті.
- `POST /admin/articles/delete/{slug}` - видалення статті.
- `GET /admin/menu-groups` - керування групами меню.
- `GET /admin/menu-groups/edit/{code}` - редагування групи меню.
- `POST /admin/menu-groups` - створення групи меню.
- `POST /admin/menu-groups/update/{code}` - оновлення групи меню.
- `POST /admin/menu-groups/delete/{code}` - видалення порожньої групи меню.
- `GET /admin/comments` - модерація коментарів.
- `POST /admin/comments/approve/{id}` - підтвердження коментаря.
- `POST /admin/comments/delete/{id}` - видалення коментаря.

## Нотатки щодо безпеки

Адмін-сторінки захищені Spring Security:

- `/admin/**` вимагає роль `ADMIN`.
- Публічні сторінки, статичні ресурси, сторінка входу та сторінки статей доступні без авторизації.
- Поточний адмін-користувач налаштовується в `application.properties` через `app.admin.username` і `app.admin.password`.

Правила:

- Не послаблювати захист адмін-маршрутів.
- Не вимикати CSRF-захист без чіткої причини та без безпечного оновлення всіх форм/API-запитів.
- Не змінювати секрети в `application.properties` без дозволу власника проєкту.
- Не комітити нові секрети, паролі, токени або приватні ключі.

## Модель даних

Основні таблиці/entity:

- `articles`
  - `id`
  - `title`
  - `slug`
  - `description`
  - `menuGroup`
  - `content`

- `menu_groups`
  - `id`
  - `code`
  - `title`
  - `sortOrder`

- `article_comments`
  - `id`
  - `articleSlug`
  - `parentCommentId`
  - `authorName`
  - `content`
  - `createdAt`
  - `status`

- `site_statistics`
  - `id`
  - `visitsCount`

Важлива деталь:

Коментарі прив'язані до статей через `articleSlug`, а не через `Article.id`.

Якщо змінюється slug статті, наявні коментарі можуть втратити зв'язок зі статтею, якщо не перенести або не оновити записи коментарів.

Не змінювати публічні URL, slug-логіку, коди груп меню або зв'язки в базі даних без дозволу власника проєкту.

## Система коментарів

Коментарі підтримують модерацію та відповіді.

Потік роботи:

1. Користувач надсилає коментар зі сторінки `/articles/{slug}`.
2. `ArticleCommentService.addComment(...)` зберігає його зі статусом `PENDING`.
3. Адміну надсилається email-сповіщення.
4. Адмін переглядає коментарі на модерації за адресою `/admin/comments`.
5. Підтверджені коментарі стають видимими на публічних сторінках статей.
6. Відповіді зберігаються через `parentCommentId`.

Публічно відображаються лише коментарі зі статусом `APPROVED`.

Відповідь приймається як дочірня лише якщо:

- Батьківський коментар існує.
- Батьківський коментар належить до того самого slug статті.
- Батьківський коментар уже підтверджений.

## Нотатки щодо Thymeleaf

Спільні layout-фрагменти:

- `templates/fragments/layout.html`
- `templates/fragments/admin-nav.html`

Контент статті відображається через:

```html
th:utext="${content}"
```

Це навмисно дозволяє HTML, збережений у контенті статті.

Обережно: це може створити XSS-ризик, якщо недовірені користувачі отримають можливість редагувати контент статей.

Створювати або редагувати HTML статей мають лише довірені адмін-користувачі.

## Статичні файли

- `static/js/main.js`
  - Відкриває та закриває секції бокового меню.
  - Показує та приховує форми відповіді на коментарі.

- `static/js/admin.js`
  - Вставляє HTML-фрагменти в редактор статті.
  - Генерує slug статті із заголовка.

- `static/css/style.css`
  - Містить стилі для layout, адмін-панелі, статей, коментарів, сторінки входу та responsive-правил.

## Нотатки щодо конфігурації

Поточна локальна конфігурація використовує MySQL:

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/java_handbook_db
```

Production-чутливі налаштування не можна зберігати захардкодженими в закомічених файлах:

- Пароль бази даних.
- Пароль адміністратора.
- SMTP-пароль.
- Облікові дані для email-сповіщень.

Для секретів краще використовувати змінні середовища, зовнішню конфігурацію або Spring-профілі.

У production бажано уникати:

```properties
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
```

Замість цього використовувати міграції та безпечні production-налаштування логування.

## Кодування

Усі source-файли, шаблони, properties, CSS і JavaScript мають зберігатися в UTF-8.

У проєкті уже є ознаки пошкодженого кодування українського тексту в шаблонах, Java-рядках і JavaScript.

Під час редагування текстів перевіряти, що українські символи відображаються коректно.

Не додавати нове пошкоджене кодування.

## Правила роботи над змінами

Перед внесенням змін:

1. Спочатку прочитати відповідні controller, service, entity, repository, template та JS/CSS-файли.
2. Показати короткий план змін.
3. Після підготовки змін показати diff.
4. Лише після підтвердження застосовувати зміни, якщо власник проєкту просить такий порядок роботи.

Загальні правила:

- Тримати зміни в межах запитаної функції або виправлення.
- Зберігати наявну структуру MVC + Thymeleaf, якщо рефакторинг прямо не запитаний.
- Залишати admin-only поведінку за `/admin/**`.
- Зберігати CSRF hidden-поля в POST-формах.
- Обережно працювати зі slug статей, кодами груп меню та зв'язками коментарів.
- Не додавати сторонні frontend-фреймворки без прямого запиту.
- Не запускати деструктивні команди.

## Правила тестування

Поточне тестове покриття мінімальне.

Після змін у коді щонайменше запускати:

```bash
./mvnw test
```

На Windows:

```powershell
.\mvnw.cmd test
```

Для змін, які зачіпають коментарі, вручну перевіряти:

- Новий кореневий коментар зберігається як `PENDING`.
- Нова відповідь зберігається як `PENDING`.
- Коментарі на модерації з'являються в адмін-панелі.
- Підтверджені коментарі з'являються публічно.
- Відповіді відображаються вкладеними під підтвердженим батьківським коментарем.
- Видалені коментарі зникають з адмінських і публічних сторінок.

Для змін, які зачіпають адмін-сторінки, перевіряти:

- Неавторизовані користувачі перенаправляються на `/login`.
- Адмін-користувач має доступ до `/admin`.
- Створення, редагування та видалення статей працює.
- Створення, редагування та видалення груп меню працює.
- Форми з CSRF-захистом коректно надсилаються.

## Відомі ризики

- Секрети зараз присутні в `application.properties`.
- Пароль адміністратора в локальній конфігурації слабкий.
- Український текст пошкоджений у кількох файлах.
- Відображення HTML статей через `th:utext` може бути небезпечним, якщо контент не є довіреним.
- Коментарі прив'язані через `articleSlug`, тому зміна slug може зламати видимість коментарів.
- Видалення статей не очевидно очищає пов'язані коментарі.
- Публічна форма коментарів не має captcha, rate limiting або захисту від спаму.
- DTO/форми не використовують Bean Validation annotations.
- Email-сервіс логує через `System.out` / `System.err`, а не через logger.
- Maven може використовувати snapshot-версію Spring Boot parent; актуальну версію перевіряти в `pom.xml`.
- Є лише базовий тест завантаження контексту.
